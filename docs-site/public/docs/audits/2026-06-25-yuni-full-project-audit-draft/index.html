0. Executive Summary

Yuni is a pnpm monorepo with a NestJS backend, Next.js frontend, Prisma/PostgreSQL persistence, Docker Compose local development setup, and documentation covering architecture, API, onboarding, and security rules.

The repository is in a generally healthy implementation state for a local MVP: backend tests, backend build, backend typecheck/lint, frontend production build, frontend lint, and Docker Compose configuration all completed successfully. The most important product blockers are cross-module integration issues rather than syntax or build failures.

Severity counts:

Severity Count
Critical 0
High 2
Medium 5
Low 4
Info 2

Highest-priority findings:

  1. Discovery requires completed profiles, but registration/profile/media flows do not set Profile.completedAt, so real users can remain invisible in discovery even after providing profile data and photos.
  2. Backend media URLs are relative /uploads/... paths. Some frontend paths normalize them against the API origin, but discovery/matches/messages use raw URLs, causing broken images when frontend and backend are on different origins.
  3. Media upload handling validates MIME/size/count but stores and serves original image bytes without image decoding, re-encoding, metadata stripping, dimension checks, or malware scanning.

Staged adapter verdict: SAFE_TO_COMMIT.

4. Findings

INT-01 - High - Discovery excludes real users because completedAt is never set

Discovery filters candidates with profile.completedAt != null. Registration creates a profile shell with completedAt: null; profile updates and media upload/update flows do not set completedAt after required profile fields or photos exist.

Evidence:

  • apps/backend/src/modules/auth/auth.service.ts: registration creates user/profile records.
  • apps/backend/src/modules/profiles/profiles.service.ts: profile update validates and stores profile fields but does not complete the profile.
  • apps/backend/src/modules/media/media.service.ts: photo upload/update changes media state but does not complete the profile.
  • apps/backend/src/modules/discovery/discovery.service.ts: discovery excludes profiles where completedAt is null.

Impact: a newly registered user can complete the visible onboarding/profile work and still never appear in discovery. This is a product-blocking integration bug.

Recommendation: define a single profile-completion rule, preferably in the profiles domain, and apply it after profile updates and photo changes inside the same logical transaction. Add backend tests for incomplete, complete-without-photo, complete-with-photo, and regression coverage in discovery.

INT-02 - High - Relative media URLs break outside profile-specific normalization

Backend stores/returns media URLs like /uploads/profile-photos/.... The profile API helper normalizes media URLs against NEXT_PUBLIC_API_URL, but discovery, matches, and messages code paths use imageUrl/publicUrl values directly.

Evidence:

  • apps/backend/src/modules/media/media.constants.ts: upload URL prefix is /uploads/profile-photos.
  • apps/backend/src/modules/media/media.service.ts: publicUrl is built from the relative prefix.
  • apps/backend/src/main.ts: backend serves /uploads on the backend origin.
  • apps/frontend/lib/profile-api.ts: profile media normalization exists.
  • apps/frontend/lib/discovery-api.ts: discovery mapping passes image URLs through directly.
  • apps/frontend/app/(app)/matches/page.tsx and apps/frontend/app/(app)/messages/page.tsx: UI consumes raw image URLs.

Impact: profile photos can work on the profile page but fail in discovery/matches/messages when frontend runs on localhost:3000 and backend serves files on localhost:4000.

Recommendation: centralize media URL normalization in one frontend helper used by all API clients, or have the backend return absolute URLs based on a configured public API/media origin.

SEC-01 - Medium - Uploaded images are stored and served as original bytes

Media upload enforces count, size, and MIME constraints, but it writes original bytes to disk and serves them statically. There is no image decode/re-encode step, EXIF/GPS metadata stripping, dimension or pixel-count validation, content sniffing, or malware scanning.

Evidence:

  • apps/backend/src/modules/media/media.constants.ts: allowed MIME types and size limits.
  • apps/backend/src/modules/media/media.service.ts: upload path writes the provided buffer and auto-approves/publishes media.
  • apps/backend/src/main.ts: static serving exposes upload files.
  • apps/backend/prisma/schema.prisma: media moderation status exists, but upload currently sets usable status immediately.

Impact: privacy-sensitive metadata can be retained in photos; malformed or polyglot files can be served as user content; oversized dimensions can create processing/display risk later.

Recommendation: add a media processing boundary that decodes, validates dimensions, strips metadata, re-encodes to a safe format, and only then persists the public object. Add tests around rejected malformed files and metadata removal.

SEC-02 - Low - Cross-site refresh/logout cookie endpoints lack explicit CSRF or Origin checks

Refresh tokens are stored in an HTTP-only cookie. In production cookie settings use SameSite=None; Secure, allowing cross-site cookie sending. The refresh and logout routes do not appear to enforce an Origin/Referer allowlist or CSRF token. Logout revokes by decoded user id when available and clears the cookie.

Evidence:

  • apps/backend/src/main.ts: CORS credentials are enabled.
  • apps/backend/src/modules/auth/auth.controller.ts: refresh/logout read cookies.
  • apps/backend/src/modules/auth/auth.service.ts: logout revokes current user refresh token and refresh validates refresh token.

Impact: impact is limited because refresh responses are protected by CORS and logout is mostly state-changing denial of session, but cross-site POSTs can still trigger cookie-backed state changes in production browser contexts.

Recommendation: add Origin/Referer checks for cookie-authenticated mutation routes or use a CSRF token pattern for refresh/logout. Verify raw refresh token identity on logout when revoking token state.

SEC-03 - Low - Rate limiting is in-memory and single-process

Endpoint-specific rate limiting is implemented in memory. This is adequate for local development and single-process MVP use, but weak for multiple replicas or restarts.

Evidence:

  • apps/backend/src/common/rate-limit/rate-limit.service.ts: in-memory buckets.
  • apps/backend/src/app.module.ts: global rate limit guard/module.
  • docs/security/README.md: docs acknowledge this as MVP-oriented.

Impact: distributed deployments can bypass limits by spreading traffic across instances, and restarts reset counters.

Recommendation: move production rate limiting to Redis, gateway-level controls, or provider WAF/rate limits before multi-instance deployment.

INT-03 - Medium - Forgot password frontend is mock-only

The forgot password page simulates success with a timeout and no backend endpoint exists for reset request or token completion.

Evidence:

  • apps/frontend/app/(auth)/forgot-password/page.tsx: mock submit behavior.
  • apps/backend/src/modules/auth/auth.controller.ts: auth controller exposes register/login/me/refresh/logout but no password reset routes.

Impact: users can enter a password reset flow that cannot send a real reset email or complete recovery.

Recommendation: either hide/label the route as unavailable until implemented or add backend reset request/confirm flows with token storage, email integration boundary, rate limits, and tests.

INT-04 - Low - Privacy and terms links point to missing routes

Auth and landing UI link to /privacy and /terms, but no corresponding app routes exist.

Evidence:

  • apps/frontend/features/auth/components/sign-up-form.tsx: links to terms/privacy.
  • apps/frontend/features/auth/components/auth-shell.tsx: footer links.
  • apps/frontend/components/landing/footer.tsx: footer links.
  • apps/frontend/app: no privacy or terms route files.

Impact: legal/support navigation produces 404s.

Recommendation: add minimal legal route pages or remove links until content is ready.

COR-01 - Medium - Blocking and match termination are not transactional

Blocking creates the block record and then separately ends active matches between the users. If the second write fails after the block is created, the system can be left in a partially applied moderation state.

Evidence:

  • apps/backend/src/modules/moderation/moderation.service.ts: block creation and active match termination are separate Prisma operations.

Impact: a blocked user might retain active match/chat state if the second operation fails, causing moderation inconsistencies.

Recommendation: wrap block creation and related match updates in a Prisma transaction. Add tests that validate both state changes occur together.

TEST-01 - Medium - No PostgreSQL integration/e2e coverage in CI

Backend tests are unit-focused and CI does not start PostgreSQL or run Prisma-backed integration/e2e tests. The schema has important relational constraints and lifecycle semantics that unit mocks cannot fully cover.

Evidence:

  • .github/workflows/quality-gates.yml: install/build/test/lint only; no database service.
  • apps/backend/jest.config.cjs: unit test configuration.
  • Existing backend specs mock Prisma interactions heavily.

Impact: relation constraints, cascades, uniqueness, transaction behavior, cursor pagination with real ordering, and migration compatibility can regress without CI signal.

Recommendation: add a focused integration test job with PostgreSQL, Prisma migrate/deploy, and a small suite covering auth/profile/media/discovery/likes/matches/moderation flows.

TEST-02 - Low - Frontend has no test/spec coverage

No frontend test files are tracked. Current frontend quality gates rely on build and lint/type checking.

Evidence:

  • rg --files apps/frontend | rg "(test|spec)\.(ts|tsx)$" returned no files during audit.
  • apps/frontend/package.json contains build/lint/dev dependencies but no test script.

Impact: auth context behavior, API URL normalization, route guards, demo/live mode transitions, and critical UI flows can regress silently as long as type checks pass.

Recommendation: add a minimal frontend test runner and start with API helper tests plus auth/profile/discovery smoke tests.

DOC-01 - Medium - Documentation is stale relative to implemented modules

Several docs still describe likes/matches/chat/moderation/notifications as future, scaffold-only, or not implemented even though backend modules and frontend routes now exist.

Evidence:

  • README.md: product status wording is behind current backend/frontend state.
  • apps/backend/README.md: module list and status are stale.
  • docs/onboarding/README.md: onboarding status lags implementation.
  • docs/architecture/foundation-hardening-plan.md: plan still frames implemented areas as future/scaffold work.

Impact: contributors can make wrong assumptions about what is safe to build on, test, or refactor.

Recommendation: refresh docs after the current media adapter work lands, especially README, backend README, onboarding, API docs, and foundation hardening plan.

DEP-01 - Info - Docker/Compose is local development only

Dockerfiles run dev commands, Compose exposes Postgres locally, and environment values are placeholders. Backend uploads are bind-mounted from the local workspace.

ARCH-01 - Info - Staged storage adapter is present but not yet wired through MediaService

The staged adapter introduces a narrow profile photo storage port and local implementation, and registers it in the media module. Existing MediaService still uses direct filesystem writes and deletes.

Evidence:

  • apps/backend/src/modules/media/media.module.ts: provider registration.

Impact: no immediate regression because existing behavior remains intact. The adapter is a foundation step, not yet a behavior replacement.

5. Staged Profile Photo Storage Adapter Review

Verdict: SAFE_TO_COMMIT.

Staged files reviewed:

  • apps/backend/src/modules/media/media.module.ts
  • apps/backend/src/modules/media/storage/profile-photo-storage.port.ts
  • apps/backend/src/modules/media/storage/local-profile-photo-storage.service.ts
  • apps/backend/src/modules/media/storage/local-profile-photo-storage.service.spec.ts

Positive observations:

  • The port is small and domain-specific: save/delete profile photos only.
  • Provider registration is scoped to MediaModule.
  • Generated filenames use UUIDs and trusted extensions based on MIME type, not original filenames.
  • Storage keys are canonicalized under profile-photos/.
  • publicUrl preserves the existing /uploads/profile-photos/... contract.
  • Delete path rejects suspicious keys through prefix, basename, regex, resolve, relative, and absolute-path checks.
  • Invalid or out-of-scope delete keys are no-ops, which is a safe behavior for cleanup paths.
  • Unit tests cover save, unsupported MIME rejection, valid delete, and invalid key no-op behavior.

Non-blocking follow-ups:

  • Add a test that simulates unlink failure for a valid key and documents whether errors should propagate or be swallowed.
  • When wiring the adapter into MediaService, verify cleanup semantics for replace/delete failure paths.

No staged-blocking security, correctness, or integration issue was found in the adapter change itself.

6. Architecture Notes

Backend boundaries are generally clear: modules own controllers/services and shared concerns sit under common. Prisma remains the central integration boundary, which is appropriate at this project size.

Current cross-module risks are mostly lifecycle coordination:

  • Profile completion depends on auth, profiles, and media but is consumed by discovery.
  • Media URL semantics are produced by backend media and consumed by several frontend clients.
  • Moderation side effects touch blocks, reports, matches, and chat visibility.
  • Likes and matches rely on expiration/rematch semantics that should eventually be covered by integration tests.

Frontend structure is understandable: app routes are thin enough, API helpers centralize most backend calls, and feature-specific components are grouped under features. The main cleanup opportunity is to remove duplicated media URL handling and separate demo/mock behavior from live API behavior more explicitly.

7. Security and Privacy Notes

Strengths:

  • Passwords are hashed with bcrypt.
  • Refresh tokens are stored hashed server-side.
  • Access token and refresh token duties are separated.
  • HTTP-only refresh cookie is used.
  • Rate-limit guards exist for sensitive endpoints.
  • DTO validation and global validation pipe are present.
  • Media upload count/size/MIME restrictions exist.
  • Moderation/report/block modules exist rather than being deferred entirely.

Main remaining concerns:

  • User-uploaded image bytes need a processing/sanitization boundary.
  • Cookie-backed mutation routes need explicit cross-site request posture.
  • Production rate limiting needs external/shared state.
  • Documentation should clearly distinguish MVP local controls from production controls.

8. Data Model Notes

The Prisma schema includes substantial domain modeling for users, profiles, media, interactions, matches, messages, reports, blocks, and notifications. The schema is more mature than several docs imply.

Important model-level observations:

  • Profile.completedAt is present and used as a discovery eligibility gate, but application flows do not maintain it.
  • ProfileMedia has ordering, type, moderation status, and isPrimary; this is a good basis for profile photo management.
  • Likes/interactions and matches encode expiry/rematch semantics that deserve DB-backed integration tests.
  • Blocks and reports provide moderation foundations, but block side effects should be transactional.

9. Frontend Integration Notes

Strong points:

  • API client files are easy to locate and inspect.
  • Profile media normalization exists in one place.
  • Auth context manages live session state and refresh behavior.
  • App shell and route structure are coherent.

Issues to prioritize:

  • Reuse one media URL resolver across profile, discovery, matches, messages, and notifications if media appears there later.
  • Replace forgot-password mock behavior with a real API flow or remove the live-looking affordance.
  • Add privacy/terms routes for links already shown in UI.
  • Introduce frontend tests around API helper normalization and auth context behavior.

10. CI and Quality Gate Notes

Current signal is good for compile-time and backend unit behavior:

  • Backend tests pass.
  • Backend build passes.
  • Backend lint/typecheck passes.
  • Frontend production build passes.
  • Frontend lint passes with warnings.
  • Compose config resolves.

Main gaps:

  • No PostgreSQL service in CI.
  • No Prisma migration validation against a real database in CI.
  • No frontend test suite.
  • Frontend lint warnings are tolerated and should be burned down gradually.

11. Recommended Remediation Order

  1. Fix profile completion lifecycle and add backend regression tests.
  2. Centralize media URL normalization or return absolute backend media URLs.
  3. Add image processing/sanitization before public serving.
  4. Make block side effects transactional.
  5. Replace or remove mock forgot-password flow.
  6. Add /privacy and /terms routes.
  7. Add PostgreSQL integration CI.
  8. Add frontend tests for API helpers/auth/discovery basics.
  9. Refresh docs to match implemented modules.
  10. Plan production storage/rate-limit/deployment hardening separately from local Compose.
